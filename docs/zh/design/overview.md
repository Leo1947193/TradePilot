# 🧩 设计文档 – 概述

## 1. 系统概述

本系统是一个**美股中期分析 Agent**，旨在针对给定的股票代码生成**结构化、以决策为导向的分析**。

系统以**股票代码作为输入**，产出**确定性结构化 JSON 输出**，包含：

- 技术分析
- 轻量级基本面分析
- 情绪分析
- 事件分析结果
- 可执行的交易计划

---

## 2. 系统目标

目标是通过强制执行**一致的分析框架**来**标准化交易前分析**，降低主观决策。

本系统：

- 是**决策支持工具**，而非自动化交易系统
- 针对 **1 周至 3 个月持仓周期**进行优化
- 专注于**可解释的输出**，而非复杂量化建模

---

## 3. V1 技术栈摘要

为保证系统以最小复杂度落地为同步、结构化的分析 API，V1 固定采用以下技术栈：

- `Python 3.11`
  - 作为统一运行时，承载分析规则、数据处理和服务逻辑
- `FastAPI`
  - 负责 HTTP API、路由组织、状态码处理和 OpenAPI 暴露
- `Pydantic v2`
  - 负责请求/响应校验，以及 graph state 和模块中间结构的模型约束
- `LangGraph`
  - 负责固定分析流程编排：`validate -> prepare_context -> parallel modules -> synthesize -> trade_plan`
- `PostgreSQL`
  - 负责持久化分析结果、交易计划和最终报告快照，支持历史追踪与复盘
- `httpx`
  - 负责访问外部行情、财务、新闻和事件数据源，并统一超时与重试封装
- `pytest`
  - 负责 API 契约测试、graph 流程测试和规则单元测试
- `uv`
  - 负责 Python 依赖管理、虚拟环境管理和开发命令执行

选型原则：

- 优先支持结构化 JSON 输出和 schema 驱动开发
- 优先保证模块边界清晰、流程可编排、行为可测试
- 优先满足 V1 的同步单接口形态，而不是为未来扩展预先引入复杂基础设施
- 工程环境与命令入口统一，避免同时维护多套 Python 环境管理方式

详细实现约束见 [../implementation/00_foundation/implementation-stack.md](../implementation/00_foundation/implementation-stack.md)。

## 4. 高层架构

系统以**数据准备 + 并行分析 + 统一决策**的流水线实现，包含：

### 4.1 数据获取层

负责收集所需输入：

- 行情数据（价格、成交量）
- 新闻与情绪来源
- 财务与估值摘要
- 事件与公司日历数据
- 宏观事件日程与公司催化剂数据

---

### 4.2 分析模块

系统将分析分解为可独立演进的模块：

- **技术分析模块**
- **基本面分析模块**
- **情绪分析模块**
- **事件分析模块**

每个模块：

- 独立运行
- 生成结构化中间输出

其中：

- **基本面、技术、情绪、事件模块**在完成数据验证与上下文准备后并行运行
- 模块之间不直接依赖彼此结论，跨模块权重组合只允许在决策层完成

事件分析模块的职责是识别持仓窗口内最重要的时间型催化剂和风险来源，例如：

- 财报窗口
- 美联储、CPI、非农等宏观敏感事件
- 公司特定催化剂，如 FDA 审批、重大诉讼、监管裁决、产品发布或并购进展

该模块应输出：

- `upcoming_catalysts`
- `risk_events`
- `event_bias`

设计原则：

- 事件模块回答的是“什么时候不该贸然开仓”以及“哪些事件会改变当前交易赔率”
- 事件模块不替代基本面、技术或情绪模块的职责
- 事件风险既可以形成方向性信号，也可以仅作为执行层面的回避条件

---

### 4.3 决策层

负责：

- 聚合所有分析模块的信号
- 处理冲突信号、缺失数据和低置信度结果
- 计算总体市场偏向（看涨/中性/看跌）

其中，事件分析在决策层的作用包括：

- 作为独立信号参与综合判断
- 在近端重大事件临近时压制可执行性
- 为最终交易计划提供明确的事件型风险约束

该层强制执行：

- 一致的决策逻辑
- 加权信号组合
- 唯一的跨模块聚合入口

---

### 4.4 交易计划生成器

负责：

- 生成可执行的交易方案
- 同时产出看涨和看跌计划

每个计划必须包含：

- 入场条件
- 止损逻辑
- 止盈逻辑

若存在近端重大事件风险，还必须明确写出：

- 等待条件
- 不交易条件

详细设计见 [trade_plan_generator/overview.md](/Users/leo/Dev/TradePilot/docs/zh/design/trade_plan_generator/overview.md:1)。

---

### 4.5 存储层

V1 固定使用 `PostgreSQL` 作为结果持久化层，负责：

- 存储历史分析结果
- 存储最终交易计划与顶层报告快照
- 支持跨时间对比
- 支持交易后评估

边界说明：

- 持久化对象是分析结果与报告，不是 LangGraph 线程状态
- 持久化用于历史检索、复盘和评估，不承担任务队列职责

---

## 5. 关键设计原则

### 5.1 结构化输出优先

所有输出必须符合严格的 JSON schema。

---

### 5.2 确定性行为

系统应最小化随机性，确保可复现的输出。

---

### 5.3 可解释性

所有结论必须可追溯至：

- 输入数据
- 中间分析
- 引用来源

---

### 5.4 模块化

每个组件应：

- 可独立替换
- 在不破坏系统的情况下演进

---

## 6. 系统边界

### 范围内

- 结构化股票分析
- 决策支持输出
- 交易计划生成
- 历史分析追踪

### 范围外

- 自动交易执行
- 高频交易逻辑
- 长期投资建模
- 复杂量化策略

---
