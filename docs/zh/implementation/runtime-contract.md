# 运行时契约

## 1. 目标

本文档定义 V1 在运行时的固定行为，用于回答这些问题：

- 哪些请求返回 `200`
- 哪些请求必须直接报错
- graph 的超时、重试和降级边界是什么
- coding agent 该如何实现同步接口

公共 API 字段仍以 [../api/openapi.yaml](../api/openapi.yaml) 为准；本文只补充运行语义。

---

## 2. API 模式

V1 固定为：

- 单接口
- 同步请求
- 同步返回完整 JSON

即：

- 客户端调用 `POST /api/v1/analyses`
- 服务端在一次请求内完成整个 LangGraph 执行
- 不返回 `job_id`
- 不提供轮询接口

V1 明确不做：

- `202 Accepted` 异步任务模式
- 后台批处理
- 流式中间事件

---

## 3. 请求生命周期

单个请求的固定阶段如下：

1. HTTP 请求校验
2. 股票代码标准化
3. 公共上下文准备
4. 四个核心模块并行执行
5. 决策综合
6. 交易计划生成
7. 顶层响应组装
8. 将分析结果与报告快照持久化到 `PostgreSQL`
9. 返回 HTTP 响应

任一阶段失败后，必须按本文定义决定：

- 直接返回错误
- 或降级继续执行

---

## 4. 超时约束

### 4.1 总请求超时

- 服务端总超时：`30s`

### 4.2 节点级超时

- `prepare_context`：`8s`
- 各分析模块节点：`12s`
- `synthesize_decision`：`3s`
- `generate_trade_plan`：`3s`
- `assemble_response`：`2s`
- `persist_analysis`：`2s`

说明：

- 节点级超时用于受控降级
- 总请求超时用于保护 API

---

## 5. 重试策略

### 5.1 可重试范围

只有外部数据获取允许自动重试：

- 网络超时
- 上游 `5xx`
- 短暂连接错误

### 5.2 重试次数

- 最多 `1` 次重试
- 使用短退避：`300ms - 800ms`

### 5.3 不可重试场景

以下情况不得重试：

- `4xx` 参数错误
- ticker 无效
- 数据结构校验失败
- PostgreSQL 写入失败
- 内部规则异常

---

## 6. 状态码边界

### 6.1 返回 `200 OK`

满足以下条件时返回 `200`：

- `ticker` 合法
- 至少形成了可组装的系统级响应
- `decision_synthesis` 和 `trade_plan` 成功生成
- 分析结果已成功持久化到 `PostgreSQL`

即使以下情况成立，也仍然可以返回 `200`：

- 某些分析模块为 `degraded`
- 某些分析模块为 `excluded`
- `actionability_state = avoid`

### 6.2 返回 `400 invalid_request`

以下情况返回 `400`：

- 缺少 `ticker`
- 请求体不是合法 JSON
- 字段类型错误
- 存在不允许的额外字段

### 6.3 返回 `404 ticker_not_supported`

以下情况返回 `404`：

- ticker 格式可解析，但不在支持市场范围内
- ticker 无法映射到受支持的标的

### 6.4 返回 `422 insufficient_data`

以下情况返回 `422`：

- ticker 合法，但无法建立最小分析上下文
- 连最基本的价格窗口、公司基础信息或分析窗口都无法准备
- 关键输入缺失导致 graph 无法进入并行分析阶段

### 6.5 返回 `503 upstream_unavailable`

以下情况返回 `503`：

- 公共上游依赖在重试后仍不可用
- 四个核心模块全部因上游不可用而失败
- `PostgreSQL` 在请求内不可用或写入超时
- 响应无法区分为用户输入问题，而明显是服务端依赖问题

### 6.6 返回 `500 internal_error`

以下情况返回 `500`：

- 响应组装违反内部契约
- 持久化数据违反内部存储契约
- graph state 非法
- 未被归类的内部异常

---

## 7. 降级规则

### 7.1 允许降级的节点

允许降级继续执行的只有四个分析模块节点：

- technical
- fundamental
- sentiment
- event

### 7.2 降级后的最小行为

节点降级时必须：

- 写入 `status = degraded` 或 `status = excluded`
- 保留失败原因到内部诊断字段
- 不伪造成功结果

### 7.3 不允许降级的节点

以下节点失败时应直接结束请求：

- `validate_request`
- `prepare_context`
- `assemble_response`
- `persist_analysis`

### 7.4 决策层降级

若进入决策层时可用模块不足：

- 仍可生成 `decision_synthesis`
- 但必须体现低置信度、低完整度和 `watch/avoid`
- 若完全无法形成系统级响应，则改走 `422/503`

---

## 8. `sources` 运行时规则

顶层 `sources` 必须满足：

- 按首次使用顺序输出
- 同一 `(type, name, url)` 去重
- 不允许输出空字符串 URL
- 不允许暴露内部临时数据源 ID

若某模块被降级：

- 仍应尽量输出已成功使用的来源
- 不得为未成功调用的 provider 虚构来源

---

## 9. 幂等性与副作用

`POST /api/v1/analyses` 在 V1 中应视为：

- 在相同数据快照下具有稳定的响应语义
- 成功请求会向 `PostgreSQL` 写入一条分析快照记录
- 不创建后台任务
- 不产生任何交易执行类副作用

补充说明：

- 同一输入在不同调用中可以产生多条历史分析记录，区别由 `request_id` 和分析时间体现
- 写库对象是分析结果快照，而不是可恢复的 graph 线程状态

---

## 10. 日志与可观测性

每次请求至少记录：

- 请求 ID
- ticker
- graph 总耗时
- 各节点耗时
- 降级节点列表
- 最终状态码

禁止记录：

- API 密钥
- 完整原始上游 payload
- 敏感认证头

---

## 11. coding agent 的默认实现准则

coding agent 开工时应默认按以下规则实现：

- 先做同步接口，不做后台任务
- 先完成受控降级，再补复杂优化
- 只对上游网络问题做一次重试
- 优先让错误边界和状态码稳定，而不是先追求全部模块都接真实数据
